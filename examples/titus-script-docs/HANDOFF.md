# AgentNebula TitusEditor-ScriptDocs — Agent Handoff Document

This document contains everything a new agent needs to continue developing AgentNebula and running the TitusEditor script documentation workflow.

## 1. Project Overview

### What We're Doing
Generating comprehensive documentation for 150+ script data types in the KF8 game engine. Each script needs:
- **_Analysis.md** (~2000-2500 lines): Deep analysis with C++ source code, KFD attributes, execution logic, real usage examples, combination patterns
- **_Reference.md** (~200-500 lines): Condensed quick reference for AI skill generation

### Current Progress
- **Task list**: `.agent-nebula/task_list.json` (152 tasks)
- **Completed**: Check `passes: true` count in task_list.json
- **Workflow status**: `python -m agent_nebula status -w .agent-nebula`

### Two Codebases
1. **AgentNebula** (`Code/AgentNebula/`): The universal workflow engine — project-agnostic, reusable
2. **TitusEditor workflow** (`.agent-nebula/`): This specific task's config, spec, tools, and state

## 2. AgentNebula Architecture

### Core Files
```
Code/AgentNebula/src/agent_nebula/
├── orchestrator.py      # Infinite loop: pick task → launch Claude session → save state
├── dashboard.py         # FastAPI web UI at :8765 (task grid, live log, session detail)
├── cli.py               # CLI: init, run, status
├── config.py            # Config management, workflow_dir/cwd separation
├── tasks.py             # task_list.json CRUD, DAG dependency resolution
├── state.py             # progress.md, session_history/, spec management
└── prompts/
    ├── initializer.py   # Phase 1 prompt (generates task_list.json from spec)
    └── worker.py        # Phase 2 prompt (executes tasks one at a time)
```

### Key Design Decisions
- **workflow_dir vs cwd**: Workflow state (.agent-nebula/) and Claude working directory are independent
- **spec.md is NOT in the prompt**: Too long for Windows CLI limits. The prompt tells the agent to Read spec.md as its first action.
- **One task per session**: Each Claude session focuses on exactly one task for atomicity
- **JSONL recording**: Every session's full conversation saved to `session_messages/session_XXXX.jsonl`
- **Dashboard**: FastAPI + WebSocket, runs in a background thread alongside the orchestrator

### Tools
```
Code/AgentNebula/tools/
├── setup_workflow.py    # Initialize .agent-nebula/ in any project
├── run_workflow.py      # Start orchestrator (auto-kills old processes, unsets CLAUDECODE)
└── stop_workflow.py     # Kill running workflow
```

### Known Issues & Solutions
| Issue | Cause | Solution |
|-------|-------|----------|
| "Claude Code cannot be launched inside another Claude Code session" | CLAUDECODE env var | `os.environ.pop("CLAUDECODE", None)` in orchestrator.py |
| "[WinError 206] filename too long" | Prompt/system_prompt passed as CLI args | Don't use append_system_prompt; tell agent to Read spec.md file instead |
| "exceeds maximum allowed tokens (25000)" | Read tool limit | spec.md instructs agent to read in chunks with offset/limit |
| Dashboard port conflict | Previous process still running | run_workflow.py auto-kills existing process on the port |
| Agent writes too short Analysis docs | No quality guidance | spec.md now has per-section quality requirements (2000-2500 lines target) |

## 3. TitusEditor Knowledge

The KF8 game engine uses a script system for character skill behavior. Scripts are embedded in timeline keyframes:

```
Role (e.g., 60001)
└── States (e.g., 101=普攻1段, 201=技能A)
    └── Keyframes (e.g., frame 0, frame 5, frame 10)
        └── Scripts (e.g., TSMoveScriptData, TSHitTestScriptData, GSPlayVisualFXScriptData)
```

## 4. Role Data Pipeline

### Binary → JSON Conversion

Raw game data is in KFData binary format (.blk/.data files):
```
Source: Code/Titus/Content/App/Data/App/Entity/Role/<roleID>/
   or:  D:/Work/Titus/Content/App/Data/App/Entity/Role/<roleID>/
```

Convert to JSON using `analyze_role.py`:
```bash
cd Code/TitusEditor/docs/DataGeneration_AI/Workflow/tool
python analyze_role.py 60001 60002 60003  # one or more role IDs
```

This generates per-role output in `Workflow/output/analyzed/`:
- `role_<ID>_full.json` — Complete deserialized data (LARGE, tens of MB)
- `role_<ID>_analysis.json` — Script statistics: per-state keyframes/scripts, per-script-type counts and samples
- `role_<ID>_report.md` — Human-readable Markdown summary

Dependencies:
- `pyKFD` library at `Code/TitusEditor/tool/KFDataTool/Py/pyKFD/`
- KFD definitions at `D:/Work/Titus/Content/EditorOnly/KFD/` and `Code/TitusEditor/tool/KFDataTool/Kungfu/ExportKFD/`
- Game data path hardcoded in script: `D:/Work/Titus/Content/App/Data`

### ExistSkillObjRef Structure

After generating analysis, copy to ExistSkillObjRef:
```bash
cp output/analyzed/role_<ID>_analysis.json ExistSkillObjRef/<ID>/
```

Current coverage (17 roles):
```
ExistSkillObjRef/
├── 60001/ — role_60001_analysis.json + individual skill JSONs
├── 60002/ through 60010/ — role_XXXXX_analysis.json
├── 60011/, 60013-60016, 60020-60021/ — role_XXXXX_analysis.json
└── ScriptUsageLookup/ — Per-script extracted usage data (see below)
```

### Per-Script Usage Extraction

The worker agent runs this tool as Step B of each task:
```bash
python .agent-nebula/tools/extract_script_usage.py <ScriptName>
```

This reads ALL `role_XXXXX_analysis.json` files and extracts every occurrence of the specified script with:
- Full parameter values
- Companion scripts in the same keyframe
- Full keyframe context (all scripts + params)
- ±3 nearby frames (timeline context)
- State metadata (ID, name, length, loop)

Output structure:
```
ScriptUsageLookup/<ScriptName>/
├── summary.json              — Cross-role stats: total_count, per_role_counts, top companions
├── 60001_<ScriptName>.json   — Every occurrence in role 60001
├── 60002_<ScriptName>.json   — Every occurrence in role 60002
└── ...
```

To regenerate all at once: `python .agent-nebula/tools/extract_script_usage.py --all`

## 5. Workflow Spec Design

The spec file (`.agent-nebula/spec.md`) evolved through several iterations to solve quality issues:

### Problem 1: Agent reading massive JSON files
- **Symptom**: Agent spent most of its session trying to parse `role_XXXXX_full.json` (tens of MB)
- **Solution**: `extract_script_usage.py` pre-extracts per-script data into small focused files

### Problem 2: Agent writing too-short documents
- **Symptom**: p3 Analysis docs were ~800 lines vs p0/p1/p2 average of 2500 lines
- **Solution**: Added per-section quality requirements with explicit line targets and content expectations

### Problem 3: Agent misreading usage counts
- **Symptom**: Agent counted JSON array items instead of using the `total_count` field
- **Solution**: Explicit instruction in spec.md to use `summary.json`'s `total_count` as the authoritative number

### Problem 4: Windows command line length limit
- **Symptom**: `[WinError 206]` when passing spec content as CLI args or system prompt
- **Solution**: Spec content is NOT injected into prompt; agent is instructed to Read the file

### Current Spec Structure
```
spec.md:
├── Goal — What to accomplish
├── What Already Exists — p0/p1/p2 docs (don't regenerate)
├── Source Code Locations — Where to find C++ headers and implementations
├── Real Usage Data — ExistSkillObjRef and ScriptUsageLookup
├── Output Requirements — Analysis (10 sections) + Reference (6 sections)
├── Task Priorities — p0 (game data scripts) → p3 (BTS/Trigger/Condition)
├── How to Execute Each Task — Step A through E with detailed instructions
│   ├── Step A: Read C++ source (with large file chunking guidance)
│   ├── Step B: Run extract_script_usage.py, read ALL per-role files
│   ├── Step C: Read format examples from p0/p1
│   ├── Step D: Write Analysis (2000-2500 lines, per-section quality reqs)
│   └── Step E: Write Reference (200-500 lines)
└── Important Rules — 7 rules for consistency
```

## 6. How to Continue

### Continue the documentation workflow
```bash
# Check current status
python -m agent_nebula status -w .agent-nebula

# Set max_sessions in config.yaml (-1 = infinite)
# Then start
python Code/AgentNebula/tools/run_workflow.py .

# Monitor at http://localhost:8765
# Stop gracefully: Ctrl+C or python Code/AgentNebula/tools/stop_workflow.py
```

### Add more role data
```bash
# 1. Generate analysis for new roles
cd Code/TitusEditor/docs/DataGeneration_AI/Workflow/tool
python analyze_role.py <roleID1> <roleID2> ...

# 2. Copy to ExistSkillObjRef
cp output/analyzed/role_<ID>_analysis.json ../ExistSkillObjRef/<ID>/

# 3. Worker agents will auto-extract per-script data via extract_script_usage.py
```

### Improve AgentNebula framework
The framework is at `Code/AgentNebula/` (git repo: github.com/jshwsw/AgentNebula). Key areas to improve:
- Dashboard: could add start/stop buttons, task reordering, manual task reset
- Orchestrator: could add parallel task execution (multiple agents working on independent tasks)
- Error recovery: currently if a session fails, the task stays pending and gets retried next session
- Cost tracking: ResultMessage has total_cost_usd but it's not aggregated across sessions

### Verify output quality
Compare new p3 Analysis docs against p0/p1/p2 benchmarks:
```bash
# Quick size check
wc -l Code/TitusEditor/docs/DataGeneration_AI/PreLearning/ScriptReadme/p3/*_Analysis.md
```
